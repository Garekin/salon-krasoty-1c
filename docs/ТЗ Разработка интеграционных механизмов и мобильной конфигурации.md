  

5. Разработка интеграционных механизмов и мобильной конфигурации

Для регистрации клиентов на услуги салона красоты разработайте мобильную конфигурацию и интеграционные механизмы для обмена с основной базой.

**Обратите внимание**

Для выполнения проекта не потребуется сборка мобильного приложения. Достаточно разработать только конфигурацию и сдать её в формате выгрузки информационной базы **.dt.*

**Требования к мобильной конфигурации:**

- Стили мобильной конфигурации должны соответствовать настольной версии конфигурации.
- В конфигурации должны быть реализованы справочники Сотрудники и Услуги (номенклатура).
- В конфигурации необходимо реализовать документ **Запись клиента**, который должен передаваться в основную базу.
- Запись клиента должна осуществляться через обработку, которая запускается при старте приложения.
- В конфигурации необходимо разработать механизм авторизации пользователя.

**5.1. Справочник «Услуги»**

В справочнике необходимо реализовать реквизиты:

| **Реквизит** | **Тип** | **Комментарий** |
|----|----|----|
| Наименование | Стандартный реквизит, длина 100 символов |    |
| Стоимость | Число (15,2) |    |

Для справочника должны быть реализованы формы элемента, списка и формы выбора с реквизитами Наименование и Стоимость.

Стандартный реквизит Код не должен отображаться в этих формах.

**5.2. Справочник «Сотрудники»**

Справочник Сотрудники необходимо реализовать с использованием стандартных реквизитов Код и Наименование (синоним — «Сотрудник», 150 символов).

В справочнике требуется реализовать формы элемента, списка и формы выбора, в которых будет отображаться только наименование (ФИО) сотрудника.

**5.3. Документ «Запись клиента»**

В мобильной конфигурации необходимо разработать документ **Запись клиента**. В документе должны быть реквизиты:

| **Реквизит** | **Тип** | **Комментарий** |
|----|----|----|
| Сотрудник | Справочник Сотрудники |    |
| **ТЧ Услуги** |    |    |
| Услуга | Справочник Услуги |    |
| Стоимость | Число (15,2) |    |

Для документа создайте форму списка, синоним формы — «Мои записи». В форме списка должны быть реквизиты: 

- Дата документа (она же — дата записи).
- Сотрудник.

**5.4. Обработка создания записей клиента**

Для создания документа Запись клиента необходимо разработать обработку.

- В обработке предполагается выбор только одной услуги. Если клиенту требуется несколько услуг, то он должен выбрать соответствующий комплекс услуг из доступного списка.
- Обработка должна запускаться после авторизации клиента в приложении.
- Обработка должна содержать реквизиты:

| **Реквизит** | **Тип** | **Комментарий** |
|----|----|----|
| Дата записи | Дата (дата и время) | Дата и время в минутах |
| Сотрудник | Справочник Сотрудники |    |
| Услуга | Справочник Услуги |    |
| Стоимость | Число (15,0) | Необходимо заполнять из реквизита Цена справочника Услуги |

- На форме обработки необходимо реализовать кнопку **Записаться**. При нажатии на кнопку должен создаваться документ «Запись клиента».
- После создания документа пользователь должен увидеть сообщение: 

  «Вы записаны на услугу <наименование услуги> к мастеру <ФИО мастера> на дату <дата с часами и минутами>».

**5.6. Разработка механизмов интеграции. Авторизация пользователя и передача документа «Запись клиента» в основную базу**

В приложении разработайте механизмы интеграций, которые позволят пользователям войти в приложение и передать документ «Запись клиента» из мобильного приложения в основную базу.

**5.6.1. Разработка механизма авторизации пользователя**

Разработайте механизм аутентификации и авторизации пользователя, который позволит проверять по логину наличие пользователя в основной базе и подтверждать либо отклонять аутентификацию.

Для этого разработайте соответствующие объекты в мобильном приложении и в основной базе:

- HTTP-сервис в основной базе должен возвращать данные только по конкретному, запрошенному пользователю. Для этого доработайте основную базу — в справочник Контрагенты добавьте поле ПользовательМП (Строка, 100), в котором будет храниться логин пользователя мобильного устройства. Из мобильного устройства логин пользователя должен передаваться в этот HTTP-сервис. Если пользователь найден по логину в справочнике Контрагенты, необходимо возвращать значение 

  ```javascript
  Истина
  ```

  . Если пользователь по логину не найден, необходимо возвращать значение 

  ```javascript
  Ложь
  ```

  .
- В мобильной конфигурации разработайте роли **Авторизованный пользователь** и **Неавторизованный пользователь**.

В зависимости от роли пользователя настройте видимость объектов: для анонимного пользователя должна быть доступна только форма авторизации, а после авторизации (после получения роли «Авторизованный пользователь») — остальные объекты (за исключением объектов для неавторизованного пользователя).

- Роль «Неавторизованный пользователь» должна быть установлена ролью по умолчанию.
- После аутентификации, если пользователь найден в основной базе, пользователю необходимо выдавать роль «Авторизованный пользователь».
- Форму авторизации необходимо реализовать в соответствии с макетами, с использованием соответствующих стилей салона красоты.

**5.6.2. Отправка документа «Запись клиента» в основную базу**

Реализуйте в приложении отправку документа «Запись клиента» в основную базу. Отправка должна происходить после заполнения формы записи — после нажатия на кнопку **Записаться**.

Для этого в основной базе реализуйте HTTP-сервис **customerOrders**:

- В основную базу требуется передавать: 
  - дату записи;
  - реквизит Сотрудник;
  - реквизит Клиент (определяется по пользователю мобильного приложения — по реквизиту ПользовательМП);
  - услугу, на которую записался клиент, и стоимость этой услуги.
- Дата документа «Запись клиента» должна быть текущей датой отправки документа.
- В мобильном приложении требуется реализовать проверку даты записи — пользователю не должна быть доступна запись на прошлые даты (при попытке создать запись на дату в прошлом необходимо изменить дату записи на текущую и выдать пользователю сообщение о том, что запись на прошлую дату невозможна).
- После нажатия на кнопку **Записаться** необходимо отправить запрос в основную базу с данными документа «Запись клиента».
- Если запрос обработан успешно, в мобильном приложении необходимо: 
  - создать и провести документ «Запись клиента»;
  - открыть форму списка «Мои записи».
- Созданный в основной базе документ необходимо провести.
- Если при отправке документа возникла ошибка, необходимо вывести об этом сообщение пользователю. В случае ошибки создавать документ «Запись клиента» в мобильном приложении не требуется.
