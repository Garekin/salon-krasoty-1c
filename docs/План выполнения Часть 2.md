## Фаза 0. Оценка сложности

**Сложность: критичная.** Признаки:
- 4 независимых направления: внешняя ПФ (БСП), подсистема бухучёта (ПВХ + план счетов + регистр бухгалтерии + проводки 4 документов + журнал + ОСВ), загрузка Excel, мобильная конфигурация с HTTP-интеграцией.
- Новый для проекта пласт — бухгалтерский учёт (субконто, корреспонденция счетов) и клиент-серверная интеграция (HTTP-сервисы, две базы).
- Неочевидные зависимости: ПВХ → план счетов → регистр бухгалтерии → доработка номенклатуры → проводки; мобилка ↔ основная база через публикацию Apache.

**Стратегия:** разбить на 5 относительно независимых блоков (P/Б/E/I/M), внутри блока бухгалтерии — строгий порядок снизу вверх по зависимостям. Тестировать в 1С:Предприятие после каждого блока. Мобильную конфигурацию вести как отдельную ИБ.

# Блок P. Внешняя печатная форма «Товарная накладная»

*Источник: Спринт 12, Тема 1; Спринт 10, Тема 3, Урок 5.*

ПФ реализуется как **внешняя обработка (.epf)** и подключается через БСП (подсистемы **Печать** + **Дополнительные отчёты и обработки**, внедрены в части 1).

> ⚠️ **Корректировка сессии 2026-06-27:** исходное допущение «конфигурация не меняется» оказалось НЕВЕРНЫМ. В этой сборке БСП для работы ПФ потребовались две доработки конфигурации — см. **P0**. Все грабли подключения подробно — в памяти `bsp-external-processing-report-assignment-gotcha.md`.

### P0. Доработки конфигурации (обязательно ДО подключения .epf)
- [ ] **Фикс формы БСП `Справочник.ДополнительныеОтчетыИОбработки.ФормаЭлемента`.** Без него регистрация любой .epf падает: «Поле объекта не обнаружено (НазначениеВариантаОтчета)». Причина: в конфигурации нет подсистемы `ВариантыОтчетов`; серверная `ЗарегистрироватьОбработку` создаёт это поле только под `ПодсистемаСуществует("ВариантыОтчетов")`, а форма (`ОбновитьИзФайлаНаСервере`, ~стр.1149) читает безусловно. Разрешить редактирование объекта (Поддержка → «Редактируется с сохранением поддержки») и обернуть чтение:
  ```bsl
  Если ПараметрыРегистрации.Свойство("НазначениеВариантаОтчета") Тогда
      НазначениеВариантаОтчета = ПараметрыРегистрации.НазначениеВариантаОтчета;
  КонецЕсли;
  ```
  Фикс одноразовый — после него подключаются ВСЕ внешние обработки/отчёты части 2.
- [ ] **Интеграция «Подключаемые команды» в `Документ.Реализация.ФормаДокумента`.** Без неё внешняя ПФ не появляется в меню «Печать»: в части 1 печать на форме сделана РУЧНОЙ кнопкой в обход БСП (модуль менеджера уже БСП-готов: `ДобавитьКомандыПечати`+`Печать()`). Добавить блоки `// СтандартныеПодсистемы.ПодключаемыеКоманды` в `ПриСозданииНаСервере` (`ПодключаемыеКоманды.ПриСозданииНаСервере(ЭтотОбъект)`), события `ПриЧтенииНаСервере`/`ПриОткрытии`/`ПослеЗаписи` и регион `ОбработчикиКомандФормы` с 4 процедурами `Подключаемый_*` (источник — Спринт 10 Тема 5 Урок 6; код — в памяти `bsp-pluggable-commands-form-integration.md`). После интеграции **убрать ручную команду «Печать»** формы и её обработчики (иначе две кнопки) — штатная «Реализация» выводится автоматически из менеджера.

2. **Идентификация клиента в мобилке→основная база:** ТЗ части 2 связывает клиента через `Контрагенты.ПользовательМП` (а не через справочник Пользователи + GUID, как в Барбершопе). В `createOrderPOST` искать контрагента по ПользовательМП = переданный логин. Передавать в JSON логин (а не GUID).
3. ~~**Регистрация нового пользователя** (метод createUser из FAQ §1)~~ — **РЕШЕНО: не делаем.** Регистрации нет в ТЗ (только авторизация по логину), а поле телефона + кнопка регистрации на форме авторизации ломают соответствие макету, которое ТЗ требует явно. Метод `createUser` основной базы и связанные мобильные элементы удалены. Если методист отдельно попросит саморегистрацию — вернуться к FAQ §1.
4. Имя файла `.dt` основной конфигурации — в ТЗ строка обрывается; уточнить точный шаблон имени у методиста перед сдачей.

        ВызватьИсключение;
    КонецПопытки;
    Возврат Результат;

> ⚠️ В основной базе обернуть `Ответственный = Пользователи.ТекущийПользователь()` в `ПередЗаписью` документа в `Попытка/Исключение` — MobileClient нет в справочнике Пользователи БСП, иначе проведение через сервис падает.

## Приложение M3. Аутентификация в мобилке (Спринт 8 Тема 12) — ФАКТ реализации

> ⚠️ `ВыполнитьАутентификацию` — **ФУНКЦИЯ, возвращает Булево**. `ОбщегоНазначения` в мобилке НЕТ → парсинг JSON через `ПрочитатьJSON`.
>
> ✅ **ПЕРЕПИСАНО в сессии 2026-07-02.** Старый вариант ниже (через `ТекущийПользователь().Записать()`) был отклонён платформой мобильного клиента («Попытка изменения информации о пользователе ИБ по умолчанию»). Рабочее решение — искать/создавать **ДРУГОГО** пользователя ИБ (не текущего) по имени = введённому логину, и назначать роль ему. Плюс находки этой сессии, без которых сценарий не проходит целиком:
> 1. **Основные роли (DefaultRoles) конфигурации = `Role.НеавторизованныйПользователь`** — обязательно, иначе пользователь ИБ без явных ролей не получит даже права зайти в клиент. После правки в Конфигураторе — **обязательно F7** (Обновить конфигурацию базы данных), иначе изменение не долетает до рабочей базы.
> 2. Работа с `ПользователиИнформационнойБазы` (поиск/создание/удаление) требует право **Администрирование**, которого нет ни у одной роли — и не должно быть (иначе неавторизованная сессия получит админку). Решение — обернуть операцию в `УстановитьПривилегированныйРежим`.
> 3. Проверка пустого логина в форме — **применена и подтверждена** (см. M6).
> 4. **Логика «Выйти» пересмотрена: не удалять пользователя ИБ, а возвращать роль на `НеавторизованныйПользователь`.** Причина — явное требование сохранять пользователей ИБ (не пересоздавать при каждом входе/выходе). Работает через `ПользователиИнформационнойБазы.ТекущийПользователь()` (а не цикл по всем пользователям, как было раньше) — логаут должен затрагивать только того, кто сейчас в приложении. Проверка `ПолныеПрава` защищает Администратора от изменения в обоих вариантах.

```bsl
Функция ВыполнитьАутентификацию(Логин) Экспорт
    Попытка
        // IP: тонкий клиент → "127.0.0.1"; AVD-эмулятор → "10.0.2.2"; реальное устройство → LAN-IP хоста.
        Соединение = Новый HTTPСоединение("127.0.0.1",,"MobileClient","mobile123");
        Запрос = Новый HTTPЗапрос("/salon/hs/users/getUser?username=" + Логин);
        Ответ = Соединение.Получить(Запрос);
        Если Ответ.КодСостояния = 200 Тогда
            ЧтениеJSON = Новый ЧтениеJSON;
            ЧтениеJSON.УстановитьСтроку(Ответ.ПолучитьТелоКакСтроку());
            Найден = ПрочитатьJSON(ЧтениеJSON);
            ЧтениеJSON.Закрыть();
            Если Найден = Истина Тогда
                Возврат Истина;
            Иначе
                Сообщить("Пользователь " + Логин + " не найден!");
            КонецЕсли;
        Иначе
            Сообщить("Ошибка аутентификации, код: " + Ответ.КодСостояния);
        КонецЕсли;
    Исключение
        Сообщить("Сетевая ошибка: " + ПодробноеПредставлениеОшибки(ИнформацияОбОшибке()));
    КонецПопытки;
    Возврат Ложь;
КонецФункции
```

> Вызов из формы авторизации (клиент, `CommonForms/ФормаАвторизации/Ext/Form/Module.bsl`):
> ```bsl
> &НаКлиенте
> Процедура ВойтиНажатие(Элемент)
>     Если Логин = "" Тогда
>         Сообщить("Введите логин пользователя!");
>         Возврат;
>     КонецЕсли;
>     Авторизован = АутентификацияИАвторизация.ВыполнитьАутентификацию(Логин);
>     Если Авторизован Тогда
>         АутентификацияИАвторизация.АвторизоватьПользователя(Логин);
>         ЗавершитьРаботуСистемы(Ложь, Истина);
> 	КонецЕсли;
> КонецПроцедуры
> ```
>
> **Актуальные процедуры** в общем модуле `АутентификацияИАвторизация` (Сервер, Вызов сервера) — работают через поиск/создание **другого** пользователя ИБ, обёрнуты в привилегированный режим (устраняет «Пользователю недостаточно прав для выполнения операции» при вызове `НайтиПоИмени`/`СоздатьПользователя`/`ТекущийПользователь`):
> ```bsl
> Процедура АвторизоватьПользователя(Логин) Экспорт
>     УстановитьПривилегированныйРежим(Истина);
>
>     ПользовательИБ = ПользователиИнформационнойБазы.НайтиПоИмени(Логин);
>     Если ПользовательИБ = Неопределено Тогда
>         ПользовательИБ = ПользователиИнформационнойБазы.СоздатьПользователя();
>         ПользовательИБ.Имя                   = Логин;
>         ПользовательИБ.ПолноеИмя             = Логин;
>         ПользовательИБ.АутентификацияСтандартная = Истина;
>     КонецЕсли;
>     ПользовательИБ.Роли.Очистить();
>     ПользовательИБ.Роли.Добавить(Метаданные.Роли.АвторизованныйПользователь);
>     ПользовательИБ.Записать();
>
>     УстановитьПривилегированныйРежим(Ложь);
> КонецПроцедуры
>
> Процедура РазавторизоватьПользователя() Экспорт // вызывается общей командой «Выйти»
>     УстановитьПривилегированныйРежим(Истина);
>
>     ПользовательИБ = ПользователиИнформационнойБазы.ТекущийПользователь();
>     Если ПользовательИБ <> Неопределено И НЕ ПользовательИБ.Роли.Содержит(Метаданные.Роли.ПолныеПрава) Тогда
>         ПользовательИБ.Роли.Очистить();
>         ПользовательИБ.Роли.Добавить(Метаданные.Роли.НеавторизованныйПользователь);
>         ПользовательИБ.Записать();
>     КонецЕсли;
>
>     УстановитьПривилегированныйРежим(Ложь);
> КонецПроцедуры
> ```
> ⚠️ **Изменено в сессии (v2):** первая версия `РазавторизоватьПользователя` перебирала `ПолучитьПользователей()` циклом и **удаляла** всех без роли ПолныеПрава. Заменено на возврат роли **только текущему** пользователю сессии (`ТекущийПользователь()`), без удаления — по требованию «пользователи ИБ должны сохраняться» (не пересоздавать при каждом входе/выходе).
>
> Команда `CommonCommands/Выйти/Ext/CommandModule.bsl`:
> ```bsl
> &НаКлиенте
> Процедура ОбработкаКоманды(ПараметрКоманды, ПараметрыВыполненияКоманды)
>     АутентификацияИАвторизация.РазавторизоватьПользователя();
>     ЗавершитьРаботуСистемы(Ложь, Истина);
> КонецПроцедуры
> ```
> (изначально было `ОткрытьФорму("ОбщаяФорма.ФормаАвторизации")` вместо перезапуска — приводило к «Недостаточно прав для просмотра», см. M6).
>
> ⚠️ **Особенность тестирования через `/N"<пользователь>"`:** запуск клиента с явным параметром `/N` жёстко фиксирует пользователя ИБ для сессии. После успешной авторизации `АвторизоватьПользователя` меняет роль **другому** (найденному/созданному по логину) пользователю, а `ЗавершитьРаботуСистемы(Ложь, Истина)` перезапускает клиент **с теми же параметрами командной строки** — то есть снова под `/N`-пользователем, у которого роль не менялась. Видимая форма входа после «успешной» авторизации — это не баг, а следствие теста с зафиксированным `/N`; для проверки результата надо перезапустить с `/N"<введённый логин>"`. На реальном устройстве `/N` не указывается, платформа сама выбирает единственного пользователя ИБ.
>
> Также нужно использовать **`1cv8c.exe`** (настоящий тонкий клиент), а не `1cv8.exe` (толстый клиент) — роли не содержат право «Толстый клиент», только Тонкий/Мобильный/Веб, иначе «Нет прав на запуск требуемого вида клиента».
